Executive summary

Nexus turns a regulator PDF into a queryable, auditable set of clause-level regulatory obligations mapped to a controlled banking vocabulary. It is a multi-stage pipeline where each stage writes its own immutable artifact and never mutates the previous stage's output.

The central design tension — and the main subject of this review — is Step 4, ontology mapping: resolving each clause's free-text concept to a stable canonical concept ID. We deliberately do not let the LLM be the final mapper. This document explains why (auditability, reproducibility, no invented IDs, cost), documents a concrete failure we observed — a PDF that mapped 100% on one run and 17.3% on a fresh run — and states the honest conclusion: the durable path is the semantic --suggest + human-approval loop, not hand-harvested exact strings.

02

Step by step — input, output, summary

Step 1

Document Acquisition & Metadata

extract_document.py

Input

  • One regulator PDF --pdf
  • Optional seeds: source-url, regulator, country, source-type

Output

  • input/<stem>.pdf (audit copy)
  • markdown/<stem>.md
  • metadata/<stem>_metadata.json
  • logs/<stem>_log.json

Deterministic text   Uses pypdf (no OCR / layout engine, so total_tables is null, not guessed). Deterministic outputs — text, hashes, page/word/char counts — never depend on the LLM. Interpreted metadata (id, dates, version, status, keywords…) is filled by one LLM call instructed to return null for anything not explicitly stated. LLM failure leaves those fields null and the run still completes.

Step 2

Structural Segmentation

structure_document.py

Input

  • markdown/<stem>.md
  • + metadata for the document id

Output

  • structure/<stem>_structure.json
  •   nested Part→Section→Subsection→Clause tree
  • logs/<stem>_structure_log.json

Mostly deterministic   A deterministic pass classifies headings by numbering convention. Only headings matching no known convention go to one LLM call (skippable via --skip-llm → defaults to clause). A stack-based pass assembles the tree by numbering depth. Known limit: table/footnote boundaries are heuristic — a node's text runs to the next heading.

Step 3

Regulatory Meaning

Fully interpretiveinterpret_document.py

Input

  • structure/<stem>_structure.json

Output

  • semantics/<stem>_semantics.json
  •   7 tags per node: domain, concept, action, object, obligation_type, mandatory_flag, confidence

The only fully interpretive step — every operative unit goes through the LLM. Controlled fields (domain, action, obligation_type) are constrained to a fixed taxonomy whose enums are stamped into the JSON schema from the merged ontology at load time, so they can't drift. concept and object stay free text — exactly what Step 4 must grapple with. Critical property: output is non-deterministic — the corpus produced 298 distinct concept strings for 314 clauses.

Step 3.5

Flat Clause Export

export_clauses.py

Input

  • semantics/<stem>_semantics.json

Output

  • clauses/<stem>_clauses.json
  •   flat records: id, no, title, text, page, semantic{}

Deterministic   Flattens the tree; every numbered provision becomes a record, structural wrappers are dropped. Domain labels → compact tokens (e.g. AML_KYC). pypdf false-positive numbers (bare years like 1934) are filtered. Flags "genuine misses" — clauses with text but no semantics — as a sign of an LLM outage in Step 3.

Step 4

Ontology Object Creation

build_ontology.py

Input

  • clauses/<stem>_clauses.json

Output

  • ontology/<stem>_ontology.json
  •   one ontology object per clause + coverage
  • logs/<stem>_ontology_log.json

Maps each clause's free-text concept onto a controlled canonical concept (stable CPT_<SLUG> ID). The one rule that makes this trustworthy: the matcher never invents a canonical ID — it only assigns IDs already in the human-owned catalog. Anything it can't match is unmapped and review_required: true. Detail follows in §03–§05.

Step 5A / 5B

Knowledge Graph & Retrieval Index

build_graph.py · build_index.py

5A → graph/

  • Node-link graph: Document / Regulator / Domain / Concept / ObligationType / Clause
  • Global node IDs → per-doc graphs union into a corpus graph; every edge carries provenance

5B → index/

  • TF-IDF index, no new deps, pluggable Backend for real embeddings later
  • Stores raw tf for a shared-corpus idf; --query runs live top-k
Step 6

Grounded Q&A

answer_query.py

Input

  • A natural-language query + 5A graph + 5B index

Output

  • answers/answer_<ts>.json
  •   answer + evidence + reasoning_path + confidence

Retrieves top-k across the corpus using a shared corpus idf (critical fix: per-doc idf made cross-document ranking invalid), walks the graph for concept/hierarchy/obligation and cross-doc "impact candidates" (same concept, other regulators), then optional LLM synthesis with a citation-grounding post-check. --no-llm degrades to structured evidence.

03

Step 4 — how the matcher works

For each clause concept, in strict precedence order — first match wins:

1exact_labelnormalized concept == a canonical label1.00
2synonymnormalized concept == a listed synonym0.95
3normalizedmatches after casefold / whitespace / trailing-punct strip0.85
4fuzzyidf-weighted token cosine over catalog surface forms + light stemming≤0.85
5unmappedno match → canonical_id null, review_required true0.00

The fuzzy tier is tried only after the exact tiers miss. Generic tokens (credit, risk, management) are down-weighted; distinctive tokens (collateral, staging, sicr) carry the signal. A fuzzy match always scores below auto_accept (0.90), so it is always routed to human review — never silently trusted.

Current review state

Everything is review_required: true regardless of confidence, because matched_concept_is_draft: true and every catalog concept is status: draft. That is intentional — see §08. Config lives in Nexus/config/step4_ontology/matching_rules.yaml.

04

Step 4 limitations — what we observed

The 100% → 17.3% collapse

Core finding

The same PDF that mapped 100% earlier mapped only 17.3% on a fresh run.

Cause: the harvested synonyms were the exact free-text strings from the first Step-3 run. A second LLM run phrases concepts differently — it produced "Credit Risk Framework Approval" and "SICR Credit Performance Indicators" where the catalog had "Credit Risk Framework" and "SICR Indicators". Exact/synonym string-matching misses the variants.

What it means: the earlier "100% coverage" was real but overfit to one run. Exact-synonym harvesting does not generalize across re-runs, because Step 3's output is non-deterministic. Hand-harvesting exact strings can never be the whole answer.

The mitigation — the fuzzy tier

What it recovered

A deterministic fuzzy fallback (idf-weighted token cosine + light stemming) absorbs phrasing variants without an LLM at run time. Fresh-run coverage 17% → 73%, with the curated CBUAE run holding at 100% (no regression). RBI docs improved 26% → 56% and 0% → 57%. Fuzzy precision is ~75–80%; because fuzzy matches are always review-flagged, wrong ones are review rejections, not silent data errors.

The honest conclusion

The fuzzy tier helps, but the durable path for Step 4 is the --suggest (semantic) + human-approval loop, not hand-harvested exact strings. Exact synonyms are a fast cache for phrasings a human has already approved; they cannot be the whole answer because the upstream (Step 3) is not deterministic.

05

Why isn't the LLM the sole / default mapper?

The LLM is very much in the loop — in --suggest (proposing catalog IDs) and upstream in Step 3. The question is why it isn't the final mapper. Four reasons, all rooted in this being a regulatory / compliance system:

1

Auditability

An auditor asks "why is this clause mapped to Credit Risk?" The matcher answers: "exact/synonym match to a human-approved catalog entry." An LLM answers: "the model decided." Only the first is defensible in a compliance review.

2

Reproducibility

We saw it live: the same PDF through the LLM twice gave different concepts, crashing coverage 100% → 17%. If the LLM were the final mapper, the obligation register would silently change on every re-run — unacceptable for a compliance record.

3

No invented IDs

The matcher can only assign IDs that exist in the approved catalog — it physically cannot hallucinate a concept. The --suggest LLM is likewise constrained to real catalog IDs; anything else it returns is discarded.

4

Cost & latency

The matcher is instant and free. An LLM call per clause per run is neither — Step 3 alone already takes ~2 min.

The role split: the LLM proposes · a human approves · the deterministic matcher decides at run time, using only approved material.

Layered ontology principle: one codebase, many regulators. A jurisdiction-neutral global layer plus a per-regulator extension, merged at runtime by nexus_ontology (an extension entry with the same key overrides the global one). Jurisdiction auto-detects from the document's Step-1 regulator metadata via registry.yaml, or is forced with --jurisdiction.

07

Where the ontology is saved

The durable, cross-document asset is the canonical concept catalog:

GlobalNexus/config/ontology/global/concepts.yaml
CBUAE ext.Nexus/config/ontology/jurisdictions/AE_CBUAE/concepts.yaml
RBI ext.Nexus/config/ontology/jurisdictions/IN_RBI/concepts.yaml  (thin)

Each entry: canonical_id (CPT_<SLUG>, stable forever), label, definition, domain, parent (hierarchy), synonyms, status (approved | draft). Everything is currently status: draft — DRAFT until a compliance expert signs off. The per-document Nexus/ontology/<stem>_ontology.json files are just the link from a document's clauses into that catalog; the catalog is the thing that must be curated and version-controlled.

08

Blockers & potential issues

Blocker
Human sign-off is the real completion gate — not code.

Every concept is status: draft, and matched_concept_is_draft: true forces 100% of mappings into review. No mapping is authoritative until a compliance expert promotes concepts draft → approved. Nothing downstream is a compliance record until it clears.

Blocker
RBI catalog is immature.

RBI docs sit at ~56–57% coverage (one at 0%). They need their own concepts + a fleshed-out IN_RBI pack, harvested via the --suggest + approval loop.

Risk
Upstream non-determinism poisons any exact-string cache.

Because Step 3 is non-deterministic, any Step-4 layer built on exact free-text strings silently decays on re-runs (the 100%→17% event). Treat single-run coverage as point-in-time, not stable.

Risk
Fuzzy precision ~75–80%.

~1 in 4–5 fuzzy matches is wrong. Contained (always review-flagged, so wrong ones are rejected) — but fuzzy coverage is not the same as trustworthy coverage. Don't report fuzzy-inflated coverage as if auto-accepted.

Risk
LLM provider fragility.

Routes through nexus_llm.call_chat with NVIDIA primary because Azure/grok returned empty HTTP 200s in streaks. If Step 3 or --suggest output is null, run python test_llm.py first to confirm provider health before assuming a code fault.

Risk
No OCR / no table structure.

pypdf means scanned PDFs yield little text (a warning is logged) and table boundaries are heuristic. Clause text can overshoot a table's extent, polluting the Step-3 concept and thus its Step-4 mapping.

Note
Coverage denominator excludes header-only clauses.

Step 4 skips clauses with no concept, so coverage_pct reflects only classified provisions. Good for signal, but cross-document comparisons must account for it.

09

Recommendation

  1. Adopt the --suggest + human-approval loop as the official mechanism for growing coverage; treat hand-harvested exact synonyms as a convenience cache only, never as the strategy.
  2. Keep the fuzzy tier (it recovers most re-run drift) but never raise its max_confidence above auto_accept — fuzzy must stay review-gated.
  3. Prioritize a compliance-expert review pass to promote the global + CBUAE catalogs from draft → approved. Until then, no output is a compliance record.
  4. Build out the IN_RBI pack next, using the same loop.
